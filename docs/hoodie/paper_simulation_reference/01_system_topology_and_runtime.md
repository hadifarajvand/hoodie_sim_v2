# 01 — System Topology and Runtime Contract

## 1. System Topology

### Paper says

The paper simulates a three-layer Cloud-Edge Continuum:

```text
IoT Area -> Edge Agent / Edge Server -> Cloud
```

There are `N` Edge Agents and one Cloud. The Cloud is represented as node `N+1`. The Edge-Agent-only set is `E={1,...,N}`.

Each Edge Agent has two roles:

1. serve tasks generated by its own IoT area;
2. host tasks offloaded by other Edge Agents.

Each task is non-divisible. A task must be placed entirely on one processing path: local, horizontal, or vertical.

### Implementation requirement

The simulator must model:

- `N` Edge Agents;
- one Cloud node;
- per-EA task source identity;
- per-task placement path;
- non-divisible task lifecycle.

### Figure 7 topology requirement

The paper includes a topology graph. Feature 090 must not assume horizontal offloading is globally unconstrained without an explicit decision. The simulator contract must record one of these modes:

- `paper_topology_adjacency`: horizontal destinations are restricted by the Figure 7 graph;
- `complete_edge_graph`: every other EA is a valid horizontal destination;
- `unknown_from_ocr_requires_pdf_visual_confirmation`: implementation is blocked until adjacency is extracted from the PDF figure.

The runtime must store the selected topology mode in every experiment artifact.

### Validation evidence

Artifacts must include:

- number of EAs;
- cloud node ID;
- adjacency matrix or explicit complete-graph statement;
- legal horizontal destinations per EA;
- rejected illegal self-offloads.

### Failure symptom if missing

If topology is not explicit, HO and MLEO behavior may be wrong, horizontal load may be unrealistic, and Figure 9/10 results cannot be audited.

## 2. Edge Agent Internal Structure

### Paper says

Each EA has:

```text
EA n
├── Private Queue n
├── Offloading Queue n
├── Public Queue 1
├── Public Queue 2
├── ...
└── Public Queue N
```

The Cloud has public queues only.

### Meaning

- Private queue: source EA's own tasks selected for local execution.
- Offloading queue: source EA's own tasks selected for offloading and waiting for transmission.
- Public queue `i` at node `k`: tasks that originated from EA `i` and were offloaded to node `k`.

Example: if EA 5 offloads to EA 3, that task enters public queue 5 at EA 3.

### Implementation requirement

The runtime must have three distinct path types:

```text
local:      arrival -> private queue -> local CPU
horizontal: arrival -> offloading queue -> horizontal link -> public queue at another EA -> public CPU
vertical:   arrival -> offloading queue -> vertical link -> public queue at Cloud -> cloud CPU
```

### Validation evidence

Artifacts must show counts and queue traces for:

- private queue insertions;
- offloading queue insertions;
- public queue insertions by source EA and destination node;
- local/horizontal/vertical path distribution.

### Failure symptom if missing

A single generic queue, direct remote service without offloading queue, or public queue without source-specific queues is not HOODIE paper simulation.

## 3. Time-Slotted Episode Lifecycle

### Paper says

The simulation is slot-based.

```text
T = 110 slots per episode
Delta = 0.1 sec per slot
```

The first 100 slots are action-taking slots. The final 10 slots drain queues and collect delayed rewards for tasks that are still pending.

### Implementation requirement

Every episode must represent:

- `slot_index`;
- action slots: `0..99` or `1..100`, consistently documented;
- drain slots: final 10 slots;
- no new action-task generation during drain unless explicitly running a non-paper test mode;
- queue progression during drain;
- reward collection during drain.

### Validation evidence

Artifacts must include:

- total slots;
- action slot count;
- drain slot count;
- arrivals per action slot;
- completions/drops during action and drain phases separately.

### Failure symptom if missing

Without the drain phase, tasks accepted/offloaded near the end of the episode may never produce reward/drop evidence. Delay and drop metrics become invalid.

## 4. Paper-Faithful Runtime Boundary

The simulator is paper-faithful only if runtime logic, not plotting logic, implements the paper. Figure files are downstream evidence, not the simulation itself.

Minimum runtime path before figure generation:

```text
slot loop -> arrivals -> action -> queue insertion -> queue service/transmission -> completion/drop -> delayed reward -> metric aggregation
```
